Local Azure development, contained

Run Azure-shaped services on your machine.

Az.Local.Dev is a self-contained Windows tray app that emulates Azure Service Bus, Storage, and Application Insights using the wire protocols official Azure SDKs already understand.

Install and run

Quickstart

Build the tray application from the Cargo workspace, then launch it from the repository root.

PowerShell
PS> cargo build -p emu-gui
PS> .\target\debug\AzLocalDev.exe
✓
Dashboard ready

The first launch opens 127.0.0.1:7777. Later, left-click the tray icon to reopen it.

02

Dashboard workflow

Create your first resource

Resource groups organize local services just as they do in Azure, without reproducing the Azure control plane.

  1. 01
    Open a resource group

    Use the default group created on first launch, or add one for your project.

  2. 02
    Add a resource

    Choose Service Bus, Storage, or Application Insights and give the instance a name.

  3. 03
    Copy connection details

    Open the information panel on the resource row. Credentials are accepted but not validated.

  4. 04
    Develop normally

    Use an official Azure SDK, Azure Functions binding, or an OpenTelemetry exporter.

SB

AMQP 1.0

Service Bus

Queues support active, scheduled, deferred, and dead-lettered messages with peek-lock semantics, lock expiry, and redelivery.

AMQP :5672+AMQPS :5671

Azure Functions binding

Set a named connection value in local.settings.json, then use that name in the trigger or binding.

local.settings.json
{
  "IsEncrypted": false,
  "Values": {
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "ServiceBusConnection": "Endpoint=sb://localhost:5672;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=emulator;UseDevelopmentEmulator=true"
  }
}

ST

Blob + Queue + Table REST

Storage

One local Storage resource exposes three path-style endpoints, matching Azurite’s port convention and the shape of an Azure Storage account.

Blob :10000+Queue :10001+Table :10002+

AzureWebJobsStorage

Use the unified connection string for Functions host bookkeeping, Blob bindings, queue triggers, Table bindings, and common Durable Functions flows.

local.settings.json
{
  "IsEncrypted": false,
  "Values": {
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "AzureWebJobsStorage": "DefaultEndpointsProtocol=http;AccountName=devstoreaccount1;AccountKey=emulator;BlobEndpoint=http://127.0.0.1:10000/devstoreaccount1;QueueEndpoint=http://127.0.0.1:10001/devstoreaccount1;TableEndpoint=http://127.0.0.1:10002/devstoreaccount1;"
  }
}

Breeze + OTLP/HTTP

Application Insights

Capture classic Application Insights telemetry or OTLP/HTTP JSON and inspect traces, structured logs, and metrics in the local dashboard.

POST /v2/trackPOST /v1/traces/v1/logs/v1/metrics

Classic SDKs

Copy the generated APPLICATIONINSIGHTS_CONNECTION_STRING value from the resource details panel.

OpenTelemetry

Use the displayed OTLP endpoint and explicitly select JSON encoding.

Environment
OTEL_EXPORTER_OTLP_PROTOCOL=http/json
i
JSON encoding only

The OTLP receiver does not accept protobuf. Configure exporters for http/json.

TokenCredential clients

Identity-style connections

Real Managed Identity requires Microsoft Entra ID and cannot be reproduced locally. Apps that create clients from DefaultAzureCredential can still connect: the emulator accepts the locally resolved token without validating it.

Trust the development certificate manually

The app asks before adding its shared self-signed certificate to the current Windows user’s trusted roots. If that step fails, run:

PowerShell
certutil -user -addstore Root "$env:APPDATA\AzLocalDev\certs\dev-cert.pem"

One process, real protocols

Architecture

The tray binary owns the dashboard, generic registry, persistence layer, and each concrete emulator engine. Client applications never need a project-specific library.

Clients

SDKAzure SDK apps.NET · Java · JS · Python · Go
FnAzure FunctionsTriggers and bindings
UIBrowserDashboard on :7777
AzLocalDev.exeSingle tray process
Service Bus engineAMQP / AMQPS
Storage engineBlob · Queue · Table REST
Insights engineBreeze · OTLP/HTTP
Dashboard + registryAxum · persistence

Crate boundaries

emu-*-coreDomain state and behavior without network I/O.
emu-*-server / amqpAzure-compatible wire protocol adapters.
emu-*-engineRunnable resource instances implementing EmulatorEngine.
emu-registryResource groups, engine factories, and generic lifecycle control.
emu-web / emu-guiDashboard APIs, embedded assets, tray app, and startup wiring.

07

Deliberately focused

Scope and limits

Built for

  • Local development and integration tests
  • Real SDK and Azure Functions workflows
  • Fast, dependency-free service setup
  • Inspecting local messages, data, and telemetry

Not a goal

  • Azure Resource Manager or billing APIs
  • Real authentication, RBAC, or Entra validation
  • Multi-node clustering or high availability
  • Complete parity with every protocol feature
